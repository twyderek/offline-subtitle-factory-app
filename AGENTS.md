# 專案協作規範

本檔是應用程式 repo 內可版本控制的治理規範；即使專案離開目前 workspace，仍必須遵守。

## 每次任務的強制前置程序

1. 執行 `npm run project:preflight`。
2. 依 `docs/project-management/README.md` 順序完整閱讀共通必讀文件，再讀取任務對應的獨立流程檔。
3. 在 `docs/project-management/08-CHANGE-LOG.md` 建立本次紀錄，先填需求、範圍、風險與驗證計畫，再修改。
4. 需求、設計、功能、部署、測試、風險或缺陷有變時，必須同步更新對應文件。
5. 完成後補齊實際修改、測試、審查、部署、風險與後續，執行 `npm run docs:check:final`。

## 開發與審查流程

1. 所有任務視為高難度工作，先制定並持續維護可執行計畫。
2. 開發完成後，啟動具獨立上下文的審查代理。
3. 審查代理只能讀取、執行與驗證，不得修改程式碼、測試或治理來源文件；唯一允許的新檔案是其本人依 `workflows/04-INDEPENDENT-REVIEW.md` 建立的本輪獨立審查報告。
4. 審查代理逐項驗證並提供證據：需求完整性、邏輯正確性、邊界情況、程式碼品質、測試覆蓋、實際運行結果。
5. 問題由主要代理修改；修正影響審查結論時要求複審。
6. 最終交付清楚區分開發驗證與獨立審查，揭露未覆蓋項目及剩餘風險。
7. 發布等級工作須先取得並記錄具體發布授權；需求方要求打包或發布，不等於接受未簽章、未公證或未實機測試等已知風險。

若文件與程式碼或發布狀態不一致，先查證後修正文檔；無法確認時標示「待確認」，不可臆測。完整責任與流程見 `docs/project-management/README.md`。
