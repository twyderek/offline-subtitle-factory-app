# 專案規範與治理

## 治理目標

- 任何版本都可追溯「需求 → 設計 → 程式碼 → 測試 → 審查 → 發布」。
- 所有高風險操作有明確授權、驗證、回復或替代方案。
- 文件反映實際狀態，不把計畫、推測或成功建置誤寫成實機驗收。

## 角色與責任

- **需求提出者／產品負責人**：決定目標、優先順序、可接受風險與發布授權。
- **主要開發代理**：規劃、分析、修改、測試、文件同步與交付；對修正負責。
- **獨立審查代理**：只讀、執行與驗證，不得修改；逐項審查需求完整性、邏輯正確性、邊界情況、程式碼品質、測試覆蓋、實際運行結果；**審查結論須以獨立檔案呈現，細節見「審查證據獨立化」一節**。
- **發布執行者**：核對版本、來源 SHA、簽章、雜湊、資產名稱、Release notes 與下載結果；**發布前須確認「發布授權」已取得且記錄完整，細節見「發布授權」一節**。

## 變更分類

| 等級 | 例子 | 最低驗證 |
|---|---|---|
| 低 | 純文字、錯字、無行為文件 | `docs:check`、差異檢查 |
| 中 | UI、規則、非關鍵 API | 相關測試、`npm run check`、人工 UI 檢查 |
| 高 | 資料寫入、媒體處理、AI、金鑰、runtime | 完整回歸、失敗路徑、封裝內容檢查 |
| 發布 | 版本、簽章、安裝包、GitHub Release | 高風險驗證＋獨立審查＋SHA／資產核對＋實機測試或揭露缺口＋**發布授權記錄** |

變更等級由執行者初步分類，但**分類本身不是最終判定**：若開發過程中發現實際影響超出原分類（例如原判定「低」但實際改動了使用者可見行為），須立即在 `08-CHANGE-LOG.md` 更正等級並註明原因，不得維持原分類走完流程後才發現。

## 階段關卡

- **需求關卡**：需求 ID、成功條件、不在範圍內事項明確。
- **設計關卡**：受影響元件、資料、隱私、安全、相容性與錯誤處理明確。
- **合併關卡**：測試通過、文件同步、獨立審查無未處理阻擋問題**且審查結論以獨立檔案存在並可查核**。
- **發布關卡**：來源可追溯、安裝包完整、簽章狀態如實揭露、雜湊一致、Release 資產可下載、**發布授權已記錄且核准範圍涵蓋本次實際發布內容（含是否未簽章、是否未實機測試等已揭露風險）**。

## 審查證據獨立化（強制）

- 獨立審查代理的結論不得僅以文字形式寫在主要開發代理維護的 `08-CHANGE-LOG.md` 內；審查代理必須產出獨立檔案（路徑規則、審查代理應填寫的格式與詳細流程見 `workflows/04-INDEPENDENT-REVIEW.md`）。
- `08-CHANGE-LOG.md` 對審查結論只能連結至該獨立檔案並逐字引用審查代理自己寫下的判定句，不得由主要開發代理重新轉述或摘要審查過程與推理依據。
- 若某條 `08-CHANGE-LOG.md` 歷史紀錄的獨立審查結論是在本條規則生效前，以轉述方式記錄、且缺乏可查核的獨立檔案，該筆紀錄不得被視為「已通過獨立審查驗證」的可信依據，須標示為歷史缺口（不得回溯偽造一份聲稱是「當時產出」的審查檔案）。

## 發布授權（強制）

- 任一變更等級為「發布」的工作項目，`08-CHANGE-LOG.md` 對應條目必須填寫「發布授權」欄位：核准人／角色、核准時間、核准範圍。
- 「核准範圍」須具體到本次發布的實際風險揭露內容（例如：是否同意在 Windows Authenticode 未簽章、macOS 未經 Apple Developer ID 公證的狀態下對外發布），而非僅有「使用者要求打包」這類籠統需求敘述——需求方要求進行打包／發布動作，不等於已核准在已知風險（未簽章、未實機測試等）下對外發布。
- 缺少發布授權記錄的既有發布，不得回溯自行推定「當時應該有獲得授權」；應標示「待確認」，並提交需求方／產品負責人補充確認或明確追認。
- 若發布授權缺失且無法及時取得需求方確認，該次發布在治理層級上視為「有條件通過」的條件之一為「發布授權待補」，不得逕自升級為「通過」。

## 禁止事項

- 未讀治理文件即開始改版。
- 用「建置成功」替代「實機可用」。
- 將 API Key、簽章密碼、token 或使用者資料寫入 repo、日誌或畫面。
- 未經需求方授權擴大外部發布、刪除、覆蓋或不可逆操作範圍。
- 審查代理直接修改檔案，或主要代理把自己的測試稱為獨立審查。
- **主要開發代理代替獨立審查代理撰寫、轉述或美化審查結論**，或以任何形式讓審查證據與主要開發代理的敘述混合在同一份由主要代理維護的文件中而不可分辨來源。
- **以「需求方要求執行某動作」（如打包、發布）推定為「需求方已核准該動作的已知風險」，並將兩者混寫為同一件事。**
- **回溯生成聲稱代表過去某個時間點產出的審查檔案、核准紀錄或測試證據**；若當時證據確實缺失，只能誠實標示缺口，不能事後補造。

## 文件與證據規則

- 每項結論應記錄日期、版本／commit、環境、指令或操作、預期、實際結果。
- 無法重現或缺乏證據時標示「待確認」或「未覆蓋」。
- 舊資料不得刪除來掩蓋錯誤；以新條目補充更正、原因與影響。
- **審查證據與授權證據須可獨立回溯來源**（檔案路徑、行號、時間戳、核准人身分），不得只以敘述性文字呈現「已審查」「已核准」而無法查核依據來源。
- **「待確認」與「待執行」語意不同，不得混用**：「待確認」指事實本身尚待人類查證或既有缺口尚未補齊；「待執行」指工作本身尚未完成。結案時（`docs:check:final`）若條目仍含「待確認」項目，須在遺留風險欄位清楚說明其性質，不得被誤判為工作未完成而卡關，也不得被誤判為已完成而忽略追蹤。
