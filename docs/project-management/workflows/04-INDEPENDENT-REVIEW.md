# 獨立審查流程

## 輸入

- 完成的差異、需求 ID、測試證據、工作紀錄與已知風險。

## 審查證據獨立化規則（強制）

- 審查代理的結論**不得**只以文字形式寫在主要開發代理維護的 `08-CHANGE-LOG.md` 內；審查代理必須產出**獨立檔案**，由審查代理本人（在獨立上下文中）建立與簽署。建立審查報告是唯一允許的檔案寫入；不得修改程式碼、測試、治理來源文件或既有審查報告。
- 獨立審查檔案路徑固定為：`docs/project-management/reviews/YYYY-MM-DD-<change-log條目slug>-round<N>.md`
  - `round1`、`round2`… 對應同一工作項目的每一輪審查／複審，不得覆蓋前一輪，保留完整歷程。
- `08-CHANGE-LOG.md` 的「獨立審查結論」欄位**只能連結**至上述檔案路徑並摘要判定結果（通過／有條件通過／不通過），**不得**由主要開發代理重新轉述審查代理的推理過程或證據內容；如需摘要，摘要文字必須逐字引用審查檔案中審查代理自己寫下的判定句，並標注來源行號或章節。
- 若某次任務因風險極低、需求方明確同意跳過獨立審查（見下方「可跳過情境」），仍須在 `08-CHANGE-LOG.md` 明確寫「本次未執行獨立審查，原因：___，需求方同意記錄：___」，不得留白或省略此欄位。

## 獨立審查檔案格式（審查代理填寫）

```markdown
# 獨立審查報告：<工作名稱>

- 審查對象 commit／版本：
- 對應 08-CHANGE-LOG 條目：<日期 — 工作名稱>
- 審查輪次：round1 / round2 …
- 審查代理啟動時間、上下文來源：（確認為獨立上下文，未沿用開發代理對話記憶）

## 1. 需求完整性
- 判定：通過／不通過／部分通過
- 證據：（引用需求 ID、實際檔案內容、指令輸出）

## 2. 邏輯正確性
- 判定：
- 證據：

## 3. 邊界情況
- 判定：
- 證據：（列出實際測試過的邊界輸入與結果，不得只寫「已考慮」）

## 4. 程式碼品質
- 判定：
- 證據：

## 5. 測試覆蓋
- 判定：
- 證據：（列出實際執行的測試指令與輸出摘要，附執行時間戳）

## 6. 實際運行結果
- 判定：
- 證據：（實際操作步驟、指令、螢幕輸出或截圖路徑；不得以「應該會成功」代替實測）

## 綜合判定
- 結論：通過／有條件通過／不通過
- 阻擋問題（若有）：
- 剩餘風險：
- 給主要開發代理的具體修正要求（若有）：

## 審查代理聲明
- 本審查代理除建立本報告外，僅執行讀取、指令執行與驗證，未修改任何其他專案檔案。
- 若上述聲明不實，本報告無效。
```

## 步驟

1. 主要代理完成開發及必要驗證後，啟動具獨立上下文的審查代理，並提供：差異範圍、需求 ID、工作紀錄連結、已知風險，**不得**提供主要代理自己對「這次改得如何」的評價性摘要，以避免預先框定審查代理的判斷。
2. 明確限制審查代理只能讀取、執行、驗證，且除新建本輪審查報告外不得修改；審查代理的執行環境應與開發環境隔離或至少可被追蹤（例如不同的 session／log）。
3. 審查代理依序評估需求完整性、邏輯正確性、邊界情況、程式碼品質、測試覆蓋、實際運行結果，並依「獨立審查檔案格式」逐項填寫。
4. 每項必須引用檔案、行為、指令、輸出或外部狀態等具體證據；**引用格式須可回溯**（檔案路徑＋行號，或指令＋時間戳），不得只寫結論不寫依據。
5. 審查代理將報告存為獨立檔案（見上方路徑規則），並在報告中做出聲明（見格式範本最後一節）。
6. 主要代理讀取審查檔案，處理問題並重跑受影響測試；主要代理的修正紀錄寫回 `08-CHANGE-LOG.md`，但**不得修改審查代理留下的獨立檔案內容**——如需回應審查意見，於 `08-CHANGE-LOG.md` 或新增的回應區塊處理，保持審查原文不被覆寫。
7. 修正影響原結論時，要求同一審查代理（同一審查角色，可為新的獨立上下文）以 `round2` 建立新檔案複審，不得就地修改 `round1` 檔案。

## 發布等級變更的額外要求

- 若對應 `08-CHANGE-LOG.md` 條目的變更等級為「發布」（見 `01-PROJECT-GOVERNANCE.md` 變更分類表），獨立審查報告除六面向外，必須額外確認：
  - 是否已取得「發布授權」欄位所載的需求方核准（見 `08-CHANGE-LOG.md` 範本）。
  - 未簽章／未公證／未實機測試等條件是否已在 Release notes 與 `00-CURRENT-STATUS.md` 如實揭露。
  - 若上述任一項缺失，審查判定不得為「通過」，最高只能為「有條件通過」並列為阻擋項，待需求方明確接受風險後才可解除。

## 可跳過情境（例外，需明確記錄）

- 僅限「低風險」等級變更（見 `01-PROJECT-GOVERNANCE.md`），且需求方在對應 `08-CHANGE-LOG.md` 條目中明確授權跳過。
- 若跳過後續發現該變更實際造成行為影響，須立即在 `07-DEBUG-AND-FIX-HISTORY.md` 建立 `BUG-ID`，並回溯執行獨立審查（round1 標記為「補做」）。

## 輸出

- 獨立審查檔案（`docs/project-management/reviews/` 下）：通過／有條件通過／不通過、阻擋問題、剩餘風險。
- `08-CHANGE-LOG.md` 內的連結與逐字引用判定。

## 停止條件

- 審查代理修改審查報告以外的檔案。
- 審查報告缺少六面向任一項，或缺少「審查代理聲明」。
- 證據無法回溯（無檔案路徑／行號／指令／時間戳）。
- 發布等級變更缺少發布授權核對。
- 阻擋問題尚未解決／未由需求方明確接受。
